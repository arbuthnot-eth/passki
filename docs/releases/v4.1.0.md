## Highlights

now owns **two IKA DWalletCaps** provisioned via real DKG ceremonies — ed25519

### 2. sol@ultron sweep — legacy → IKA-native, on-chain verified

Drained the raw-keypair-derived `7iVxCjQpLEhsYTLCUVJkPBZvnTXgtLKmNLKVDMrctv3U`
to the new IKA-native address in one server-orchestrated sweep:

- **0.0119 SOL** (SOL transfer post rent refund)
- **136.47 USDC** (`EPjFWdd5…zybapC8G4wEGGkZwyTDt1v`)
- **90.75 iUSD SPL** (`Jk4P1ADUyiEY9e6X4VRPt9vN8Za87tjZ7sq2QWRgpps`)

Transfer + closeAccount in one PTB per mint via new instruction builders
`transferCheckedIx`, `closeAccountIx`, `systemTransferIx`, and the
`sweepSplAccount` / `sweepNativeSol` helpers in `src/server/solana-spl.ts`.
`POST /api/cache/sweep-sol-ultron` is admin-gated via a signed personal
message.

### 3. WASM-in-DO feasibility proven — blocker memory falsified

`@ika.xyz/ika-wasm` loads and runs inside a Cloudflare Durable Object
runtime. Host imports bind cleanly (only `crypto.getRandomValues`,
`String`, `Array`, `Error`, plus `wbindgen_*` internals — all
Workers-safe). New `UltronSigningAgent` DO class with:

- `/api/ultron/wasm-spike` — sanity test, calls
  `generate_secp_cg_keypair_from_seed(0, seed)` in <1 ms
- `/api/ultron/read-dwallet` — reads ultron's dWallet via
  `IkaClient(SuiJsonRpcClient)`, 88–568 ms round trip
- `/api/ultron/accept-share` — transitions the dWallet from
  `AwaitingKeyHolderSignature` → `Active` via `acceptEncryptedUserShare`,
  all server-side: derives deterministic seed, fetches `user_public_output`
  from the DKG tx event log, builds PTB via IkaTransaction, signs with
  ultron's keypair, submits via JSON-RPC

**Two IKA SDK blockers beaten along the way:**

1. The SDK's `wasm-loader.js` detected `process.versions?.node` (true under
   `nodejs_compat`) and tried to use the Node variant which wants
   `fs.readFileSync`. Fix: `src/server/wasm/ika-wasm-shim.ts` eagerly runs
   `initSync(bakedModule)` and re-exports every WASM function on a
   `default` object, with `alias` in `wrangler.jsonc` pointing
   `@ika.xyz/ika-wasm` at the shim.

2. `acceptEncryptedUserShare` wants the user's *centralized* DKG output,
   which isn't stored on-chain — only the network's decentralized output
   is. Fix: fetch the DKG tx via `sui_getTransactionBlock showEvents=true`
   and read `event_data.user_public_output` from the `DWalletDKGRequestEvent`
   (232 bytes for ed25519).

### 4. Vector-principles intent authentication — helper extracted + reused

`src/server/vector-intent.ts` centralizes the five Vector principles from
the existing `bam-mint-v2` endpoint (digest-bound intent, relayer/authority
split, hash-chain state progression, no pre-reveal, expiration primitive).
`bam-mint-v2` now calls the helper. New `POST /api/cache/send-iusd-v2`
endpoint reuses it to wrap the existing `send-iusd` flow. Client-side
signer helper at `src/client/vector-intent.ts`.

### 5. Deterministic encryption seed for ultron DKG

`/api/cache/rumble-ultron-seed` — admin-gated endpoint (signed personal
message from an allowlist) that returns
`sha256(SHADE_KEEPER_PRIVATE_KEY ‖ "ultron-dkg:<curve>:<ultronAddress>")`.
Browser `window.rumbleUltron()` fetches the seed before running DKG so the
same encryption keys can be re-derived later by any server-side process
that knows the keeper key. Used during DKG, and by `accept-share` in this
release to reconstruct the encryption keys for the on-chain accept step.

### 6. User-funded StableShadeOrder<iUSD>

Shade orders now route through the user-signed StableShadeOrder<iUSD>
path when brando has enough iUSD — they own the on-chain order directly
and cancel refunds flow back to them naturally without the
forward-to-holder hack. New client builders in `src/suins.ts`:
`buildCreateStableShadeOrderTx`, `buildCancel{,Refund}StableShadeOrderTx`,
`findCreatedStableShadeOrderRef`. `_shadeCreate` prefers the user path,
falls back to ultron shade-proxy only on insufficient iUSD.

### 7. Helius proxy — kills client 403 spam

`POST /api/sol-rpc` relays an allowlisted JSON-RPC method set
(`getBalance`, `getAccountInfo`, `getTokenAccountsByOwner`, etc.) to
`mainnet.helius-rpc.com` using `HELIUS_API_KEY` server-side. Client
`SOL_RPCS` flipped to `['/api/sol-rpc', 'solana-rpc.publicnode.com']` so
the browser talks same-origin and stops hammering
`api.mainnet-beta.solana.com`.

### 8. ULTRON_SOL_ADDRESS constant — 13 address-lookup sites

Every balance lookup, deposit target, webhook matcher, and sub-cent tag
derivation in `src/server/agents/treasury-agents.ts` now references
`ULTRON_SOL_ADDRESS = 'GfVzGHiSPyTnX6bawnahJnUPXeASF6qKPd224VQws1DW'`
instead of recomputing `base58(keypair.getPublicKey().toRawBytes())`. Four
raw-pubkey usages remain and are legitimate mint-authority paths — they'll
be migrated when iUSD SPL mint authority transfers to the new address
alongside DO-hosted signing.

### 9. Semver-compatible dep refresh

13 packages bumped (patch/minor only), including `@mysten/sui` 2.14→2.15,
`@mysten/suins` 1.0.2→1.0.4, `@noble/{curves,hashes}` 2.0→2.2,
`@mysten/deepbook-v3` 1.2→1.3. Held back three major bumps
(`@bluefin-exchange/bluefin7k-aggregator-sdk` 5→6, `agents` 0.6→0.10,
`typescript` 5→6) for separate validation passes.
